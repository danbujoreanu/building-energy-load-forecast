# ADR-009: City-specific Processed Data Paths to Prevent Cross-city Overwrites

**Status:** Accepted
**Date:** 2026-03-15 (Session 30)

---

## Context

The pipeline originally wrote all preprocessed data to a single shared `data/processed/` directory. This became a critical silent-corruption bug when both the Drammen and Oslo pipelines were run sequentially: executing `python scripts/run_pipeline.py --city oslo` overwrote the Drammen `model_ready.parquet`, `splits/` directory, and `scaler.pkl` with Oslo-derived versions. The Drammen processed data was permanently lost and had to be regenerated from source.

The failure mode was particularly dangerous because it was completely silent — no error was thrown, the Oslo run reported success, and the corruption was only discovered later when Drammen evaluation results changed unexpectedly. Any pipeline invocation that writes to `data/processed/` without a city qualifier will clobber artifacts from any previously processed city.

---

## Options Considered

1. **Keep shared `data/processed/`** — simple directory layout, but fundamentally cannot support multi-city operation. Any two-city run destroys one city's artifacts. Non-starter for a multi-city research pipeline.

2. **City-specific subdirectory: `data/processed/{city}/`** — each city owns an isolated subtree. Drammen artifacts live at `data/processed/drammen/`, Oslo at `data/processed/oslo/`. Parallel city runs are safe. Clear ownership of processed artifacts at a glance. Minimal change to config and scripts.

3. **Timestamped directories: `data/processed/{city}/{timestamp}/`** — full historical artifact trail (useful for debugging regressions). However, this requires a resolution mechanism (which timestamp is "current"?), adds symlink or config management complexity, and doubles or triples storage consumption for large multi-building parquet files. Overkill for a two-city research pipeline.

---

## Decision

**Option 2 — city-specific subdirectory (`data/processed/{city}/`).**

This is the minimal change that eliminates the silent-overwrite failure mode. It adds no operational complexity: `cfg["paths"]["processed"]` remains the base path in `config/config.yaml`, and all pipeline scripts append `/ city` at runtime. The layout is immediately legible in a directory listing.

---

## Consequences

**Positive:**
- Drammen and Oslo pipelines can now run concurrently or in any order without data corruption. Each city's `model_ready.parquet`, `splits/`, and `scaler.pkl` are fully isolated.
- Clear ownership of processed artifacts: the directory name is the city name. No ambiguity about which model-ready file corresponds to which city.
- Consistent layout across all three pipeline entry points that write processed data.

**Migration required:**
Any `data/processed/` artifacts written before this change (without a city subdirectory) are structurally incompatible with the new path expectations. They must either be moved manually into `data/processed/{city}/` or regenerated by re-running the relevant pipeline script. In practice, the Oslo processed data was migrated; the Drammen processed data was lost in the incident and regenerated from source.

**Config convention:**
`cfg["paths"]["processed"]` in `config/config.yaml` remains as the base path (e.g. `data/processed`). All pipeline scripts that write processed data construct the full city path as:

```python
proc_dir = Path(cfg["paths"]["processed"]) / city
proc_dir.mkdir(parents=True, exist_ok=True)
```

**Files updated in Session 30:**
- `scripts/run_pipeline.py` — primary pipeline; `proc_dir` now `data/processed/{city}/`
- `scripts/run_raw_dl.py` — Setup C (PatchTST); same fix applied
- `scripts/run_horizon_sweep.py` — Sprint 2 horizon sweep; same fix applied

All three scripts were updated in the same session commit to ensure no script retained the shared-path behaviour.
